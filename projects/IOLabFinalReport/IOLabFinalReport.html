Intro / Motivation

If you experience an uncertainty about the food you are about to order at a restaurant, specifically if it could contain a serious food allergen, then you are one among approximately 15 million Americans affected by food allergy. In addition to this, there are many who have specific dietary preferences arising out of lifestyles, religious beliefs as well as personal preferences. Of the serious food allergies, the top 8 ingredients that cause them - milk, eggs, soy, wheat, shellfish, peanuts, fish and treenuts - are widely used as ingredients in a lot of foods we eat in restaurants.

To consider the needs of this population, information on restaurant foods and what they contain are available online, however three key information problems still exist. First, the information with ingredients and what they themselves might contain can be extremely detailed and complex. Second, the information that currently exists is static and is not tailored to individual user preferences/behaviors which is the origin (and the beauty) of the question in the first place. Thirdly, even with the existence of both information systems (food ontology and user preferences), there is no usable interface which bridges these two systems and presents it to the user at the time of need with little hassle.

This is what motivated us about building Koàlacarte!

On closely inspecting the third problem, there was an additional opportunity here as having a solution on the move can not only help you decide what to eat at a certain restaurant, but also make restaurant selections based on location and of course, dietary preferences. By utilizing the individual characteristics and architectures of each information system and translating them into databases and mobile applications, we sought to weed out both dishes and restaurants that do not belong in the dietary preferences of users and help them explore, find and choose where and what to eat.

An additional opportunity here was to fully utilize the team composition and apply individual strengths to various aspects of this problem. Nico worked on creating the backend logic on Ionic in addition to modelling the databases (derived from project Foodwise which Nico and his team worked on in their applied NLP class) and UI engineering. Kinshuk focused on integration of the MLab Mongo API in addition to working on the Ionic backend logic with Nico. Ganesh focused on the user experience of the app by working with the design team on Nico’s MIMS Final Project group and used that to help model the database in addition to working on front end code.

In addition to implementing some of the theoretical aspects (like ontologies) of Information Organization and Retrieval class, the key lessons from Information Organization laboratory that we used for our solution were working with front-end frameworks and architectures, using good database practices to model our database and synergistically inform the UX as well as using APIs and using the information from our class to choose a mobile friendly framework to build our solution.

What is KoÁlacarte

KoÁlacarte is a Service that allows user to identify dishes that are safe to eat in a restaurant keeping in mind dietary restrictions and  preferences.

The problem space that the service address is food and dining industry. It aims to provide diners and especially those with special dietary needs. Such a service will most likely be used when a user is sitting in restaurant and has to make decision about the item to order from the menu. The service must make this process easy and stress free for its users. For this reason the it was important to make the service available portable rather than desktop based.

The heart of the application lies in its rich database. It uses Mongo document database which holds information about the two main entity involved in the database model.

The first entity is the user. The database holds all the information about user preferences and restriction as well as the information about dishes he likes and saves in different restaurant.

The other entity are the restaurant. At phase of development the database contains a comprehensive list of all the restaurants in San Francisco area. Each restaurant's has information such as its geo location, street address, cuisine, etc. It links to different menu that the restaurant offers. Each menu has different subsection which in turn has menu items. For eg a restaurant called “Harvey’s” has menus namely, “Eat”, “Dinner”, “All day”, “Brunch”, Dinner has subsection, “Mains”, “Starts”, “Soup and Chili”, Mains the has other items listed under it.

The value addition is done by analysing millions of food item menu to extract common ingredient found in them. These ingredients are listed under each menu item with a number denoting its probability in the dish. This data is then matched up with user dietary data to explicitly mark out dishes that has a probability of containing an ingredient that might trigger dietary concern for the user. The application marks out the dishes in a visually pre attentive manner that takes the eases the burden of choice while deciding what to eat for the user.

The application uses hybrid database to manage information and an API layer to delegate information retrieval and storage in the database. Apart from the Mongo database. Application makes use of Firebase database. This is especially designed database optimized for maintaining user information such as user id, user name, email and password. It takes care of user authentication, password change and retrieval, in a secure manner. This is used in conjunction with Mongo database to manage user information. All communication of the application to Mongo database happens via an API layer. The API layer is created via MLab API which provides a Restful HTTP API. All the information from database is queried by the application via a GET request to the API. All the information is also posted to the API via POST request.

Details of solution - Ionic

Ionic

Keeping in mind the use case of the service it was important to implement mobile version of the application. Therefore we decided to implement the application using Ionic on top of cordova framework. Ionic allows us to easily build native phone apps for various mobile operating system. The code is written using HTML5, Javascript, AngularJs and CSS and ionic produces the automatically produces native application  for mobile platform.

Working of Ionic

The application works like a single page application with just one html page rendering at all times. The transition from one screen to another is done via templates and views. Each screen is made into a template and on transitioning to the screen the template is plugged into the main html page and the dependences are “injected” into the page thanks to Angular’s architecture. Transition is managed by routes.js file. This file defines each template as a state, adds controller  and template url.  Each template has its own controller defined. It is possible to have different controllers for different parts of the page. All the variables that are used in template is defined and assigned in its controller. All the controllers are written in the controllers.js file.

Functionality that is available across all controller are called factories. There could be multiple factories logically divided by the work it performs. All the controllers can access any of the factories. These factories are defined in services.js. App.js initializes the app. Directive.js is used to define directives which can bind to individual DOM element and can be used to manage them. Angular also has many built in directives that can be used without defining. Additionally, HTML has a localStorage function that allows you to store up to 5MB of strings and persist in the cache of the application. It’s ease of use is notorious but it’s used sporadically as to not hit the 5MB cap and limit functionality.

Workflow

The application supports 2 main workflows.

  1. User uses the app for the first time.
  1. User is greeted by the welcome page where he can sign up by clicking on sign up link
  2. On the sign up screen, user has to enter his name, email and password. This information is saved in firebase by making a call to firebase database in user authentication factory. User preferences (blank at this point) and an identifier is also stored in document database using mlab API by making a post HTTP request. This request is made in api call factory in services.js file.
  3. The user now goes to success page where he has the option to look at restaurants, select food preferences or go to profile.
  4. On selecting “select food preferences”, the user is taken to a page where he can select from a list of all the ingredients the ones that he is allergic to. This information is transmitted to database via mlab API
  5. User can then decide to look at restaurants near him.
  6. User can select a restaurant and browse menus and sections. The menu item that might contain the ingredient that the user is allergic too is marked using a red icon. User can select the item to look at the ingredients and the possibility of its existence in the dish.

  1. User returns to app.
  1. The user information is already stored for the user so when user returns to app he directly taken to restaurant search page
  2. User can look inside the restaurant as before
  3. User can go to his profile and make any updation or addition as needed.

Conclusions

The main takeaway from this project is how current technologies allow you to put together a complex multi-layered application from start to finish within a few months. It helped break down any self-doubt of embarking on a new project involving technologies or languages that are unfamiliar or even daunting.

On a technical level, one of the main lessons is the advantages of using a framework with rich testing features and a wealth of components to use rather and creating many of the pieces of your application from scratch. Ionic provided a number of UI and data management components that made some of the more complicated pieces of the project more manageable.

Also, on a data intensive projects such as this one processing and cleaning of data was the most arduous and time consuming part. On the other hand, once the cleanup has been done a number of opportunities become apparent that were harder to visualize. For instance, we always envisioned a user's preferences being saved in the database to be persisted and retrieved for use to decide what dished to flag for users but we never considered the option of having users adding more ingredients and dishes to that list of undesirable foods on fly as they browse through menus. That is now in our list of upcoming features for the next version of the application.

Lessons Learned

Another insight came to us when we encountered difficulties adding user creation and authentication features to the application. There were many variables that we did not account for while planning the feature and implementing it would have drained much of our time, taking resources away from building our main features. Including a new technology into your application stack can be time consuming, but if you pick pick a low-friction ready to go tool it might save you more time that it will take you to build the functionality the tool will be performing. Future Direction

The future direction of KoÀlacarte involves using the full capabilities of mobile devices as well as expanding functionality based on user input and user usage patterns.

One of these features will involve using the device’s GPS to only show you restaurants near your location. At this point there are two constraints that prevent this functionality from happening: (1) the actual code that makes use of the location API in the device, which Cordova and Ionic should be able to access, and (2) enough pre-processed restaurant data so that the odds of you finding a restaurant close to your location are high (as long as you are in San Francisco that is. More long term, we would like to add more cities within the Bay Area and beyond.

Another promising possibility for this application is the ability to plan in advance a series of meals that match your eating preferences and to integrate with an online ordering application to have those meals delivered to you automatically without having to interact with two or three apps and/or making a call every day.
